Cow Abductor

A top-down arcade game built using only the simple Tmpl8 library, which handles rendering and tick logic.

Overview

Cow Abductor is an arcade game built in C++ using the Tmpl8 2026 framework, submitted as my intake assignment for BUas CMGT.

My first C++ project

This project was my first ever exposure to C++ and engine-less gamedev, and the transition from C# and Unity took some time. Worrying about memory management, circular dependencies, and the complete lack of engine tooling made every new system a labour of love, but at the same time the newfound level of control and ownership over my game was gratifying.

The GameObject system

Instead of opting for a deep inheritance hierarchy, I drew on my familiarity with Unity to design a composition-based system, where a gameObject's behaviour is defined almost entirely by the components it owns.

C++
class GameObject
{
public:

    Tmpl8::vec2 pos;
    bool debug = false;

    void Start();
    void Tick();


    template <typename T, typename... Args>
    T& AddComponent(Args&&... args)
    {
        auto comp = std::make_unique<T>(std::forward<Args>(args)...);
        comp->gameObject = this;
        T& ref = *comp;
        components.push_back(std::move(comp));
        return ref;
    }

    template <typename T> T* GetComponent()
    {
        for (const auto& component : components)
        {
            T* ptr = dynamic_cast<T*>(component.get());
            if (ptr) return ptr;
        }
        return nullptr;
    }


    void SetActive(bool isActive);

    GameObject(Tmpl8::vec2 spawnPos, bool debug) : pos(spawnPos), debug(debug){};


private:

    std::vector<std::unique_ptr<Component>> components;

    bool active = true; // Whether to run Tick() logic

    void DrawOrigin();

};

The base GameObject class is barebones, containing only a vector2 position, component-handling logic, and members for lifecycle and debugging. At the core of this is the AddComponent() function, which uses variadic templates to pass any number of Component constructor arguments through a single Args... parameter.

The window system

When the player drags a window, all entities inside it move with it while maintaining their relative positions. Each entity stores a draggingOffset at the moment dragging begins, and its world position is recalculated from the window's current position during the drag. Bringing a window to the front is handled by removing it from the windows vector and pushing it to the back, since draw order is back-to-front.

C++
void Window::UpdateDrag(vec2 mousePos)
{
    transform.position = mousePos - draggingOffset;
    transform.position.x = Clamp(transform.position.x, 0.0f, 1024 - (transform.scale.x + 1));
    transform.position.y = Clamp(transform.position.y, 0.0f, 768  - (transform.scale.y + 1));

    for (int i = 0; i < entities.size(); i++)
        entities[i]->transform.position = transform.position - entities[i]->draggingOffset;
}

Enemy separation was handled with a proportional repulsion force based on overlap depth rather than hard collision resolution, which avoids jitter when more than two objects are involved.

The collision system

The collision system is built around SAT (Separating Axis Theorem) and supports polygon vs polygon, polygon vs sphere, and sphere vs sphere. Every overlap test returns a Minimum Translation Vector which is the shortest vector along which overlapping shapes can be separated. This MTV drives both collision resolution and the repulsion system.

C++
bool Collider::CheckOverlap(const Collider& other, MTV& outMTV) const
{
    float smallestOverlap = FLT_MAX;
    vec2 smallestAxis;

    for (const vec2& localAxis : localNormals)
    {
        vec2 axis = transform->TransformDirection(localAxis);
        Projection a = ProjectOntoAxis(localVertices, axis, *transform);
        Projection b = ProjectOntoAxis(other.localVertices, axis, *other.transform);

        if (!Overlaps(a, b)) return false;

        float overlap = std::min(a.max, b.max) - std::max(a.min, b.min);
        if (overlap < smallestOverlap) { smallestAxis = axis; smallestOverlap = overlap; }
    }

    outMTV.axis  = smallestAxis;
    outMTV.depth = smallestOverlap;

    if ((transform->position - other.transform->position).dot(outMTV.axis) < 0.0f)
        outMTV.axis = -outMTV.axis;

    return true;
}

The window border colliders use CheckFullOverlap a variant that checks whether a sphere is completely contained within a polygon rather than just touching it. This ensures entities only transfer windows when fully inside the overlap zone, not just grazing the edge.

Reflections

This project pushed me into a lot of unfamiliar C++ territory. forward declarations to break circular includes, constructor initializer lists, virtual function chains across multiple inheritance layers, and pointer ownership. Most of these came up as real problems during development rather than things I planned for upfront, which made solving them feel meaningful.

The window transfer mechanic ended up being more emergent than designed. Once the system was working I realized the player could actively use window manipulation as a combat tool, dragging a window to steal enemies, or repositioning windows to funnel bullets. That kind of emergent gameplay from a simple rule set is something I want to keep exploring.
